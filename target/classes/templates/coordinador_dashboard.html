Conclusión

Interpretación y reflexión del modelo

Al resolver el modelo mediante el método Simplex LP en Excel Solver, se encontró que la solución óptima es procesar 300 registros de asistencia al día (X₂ = 300) y dejar en cero la gestión de clases independiente (X₁ = 0), alcanzando una eficiencia de Z = 1800 puntos.

La restricción de tiempo es la limitante clave. Con 300 minutos disponibles y cada asistencia consumiendo 1 minuto, el sistema puede procesar exactamente 300 registros antes de agotar su tiempo operativo diario. Esta restricción es vinculante (no tiene holgura).
Gestionar clases no es rentable bajo estas condiciones. Una clase consume 3 minutos y aporta 10 puntos. Una asistencia consume 1 minuto y aporta 6 puntos. La razón eficiencia/tiempo para clases es 3.33, mientras que para asistencias es 6. El modelo reconoce que maximizar registros de asistencia es la estrategia más eficiente.
El sistema Aula está correctamente dimensionado. Las restricciones de procesamiento (holgura de 100 ops) y validación QR (holgura de 50) indican que esos recursos no son el cuello de botella. Si se quisiera mejorar el desempeño, la inversión debe dirigirse a ampliar el tiempo operativo del servidor.
El modelo valida el diseño del proyecto. El sistema fue construido para automatizar el registro de asistencias masivo, y la IO confirma matemáticamente que esa es la función que maximiza el valor del sistema bajo restricciones reales de infraestructura universitaria.
